# Release v0.1.21

**Signed full-policy handoff for read-only release CI.**

## Overview

v0.1.21 is a corrective release for the version-tag publication gate introduced
in v0.1.20. It changes release mechanics and their documentation only. No schema,
contract, capability token, or engineering-principle obligation changes.

The v0.1.20 signing ceremony and tag protection worked: the tag was signed by
the pinned key, GitHub verified it, the creation restriction was evaluated, and
the organization-administrator bypass was accepted. Publication then failed
closed before creating a release because CI attempted to repeat an assertion it
could not observe.

GitHub returns a repository ruleset to read-only callers, but deliberately omits
`bypass_actors` unless the caller can write that ruleset. The standard
`GITHUB_TOKEN` had repository metadata read access, so every public ruleset field
was visible while the bypass list was redacted. Treating the redacted list as an
empty configured list produced a false policy-drift failure.

The signed v0.1.20 tag is preserved as an immutable record and is not moved or
recreated. v0.1.21 carries the repaired workflow and becomes the published
corrective release.

## The Three-Part Publication Gate

### 1. Full Pre-Tag Assertion

`make release-tag` now performs the complete ruleset check inside the release
script before creating the tag. The maintainer credential must be able to see
the ruleset's bypass actors. The guard requires exactly:

- repository ruleset `Tag Publish Protection`, active for tags;
- inclusion of only `refs/tags/v*`, with no exclusions;
- creation, update, deletion, and non-fast-forward protections;
- a sole `OrganizationAdmin / always` bypass.

Directly invoking `release-tag.sh` no longer bypasses this check; the script owns
the safety invariant rather than relying only on a Make prerequisite.

### 2. Signed Full-Policy Handoff

After the full check passes, the guard canonicalizes the expected policy and
computes its SHA-256 fingerprint. `release-tag.sh` embeds this line in the
annotated tag message before signing:

```text
Tag-Publish-Policy-SHA256: <canonical-policy-fingerprint>
```

The pinned-key signature covers the tag target, release intent, and policy
fingerprint together. CI can therefore verify the maintainer's complete
admin-visible assertion without receiving the credential that made those fields
visible.

### 3. Read-Only Publication Assertion

Release CI uses `--read-only` to verify every ruleset field available to its
standard token. Omitted or empty bypass actors are accepted only as the API's
documented redaction. If bypass actors are visible, they must match the complete
policy; an unexpected visible actor still fails.

After the pinned-key signature passes, CI requires the exact expected policy
fingerprint inside that same tag object. A missing or incorrect attestation
fails before publication.

This split keeps CI honest and least-privileged: it checks what it can observe,
verifies a signed handoff for what it cannot, and never gains ruleset-write
authority merely to inspect the bypass list.

## Why v0.1.21 Publishes Differently

The corrective tag points at the commit containing this repaired workflow.
When v0.1.21 is created through `make release-tag`:

1. the full live policy must pass;
2. the signed tag receives the canonical policy fingerprint;
3. CI accepts the documented read-only bypass redaction while still checking
   all visible policy fields;
4. CI verifies the fingerprint only after the pinned-key tag signature passes;
5. exact tag-object binding and the final pre-publication ref check remain
   unchanged.

A version bump alone would not repair the path. The new tag must be created from
the commit containing these control changes.

## Negative Controls

The release-control test suite now proves:

- full mode rejects omitted or empty bypass actors;
- read-only mode accepts omitted, empty, or exact bypass data;
- read-only mode rejects an unexpected visible repository-role bypass;
- wrong ref patterns, missing protections, extra rules, and disabled enforcement
  fail in both publication paths;
- missing or incorrect policy attestations fail.

A pre-release manual negative check also confirms that the existing v0.1.20 tag,
which predates the attestation design, is rejected by the new attestation
verifier.

## Adoption Guidance

Repositories adopting this release pattern should keep the three controls
together:

1. run a complete pre-tag guard with a maintainer credential that can view all
   policy fields;
2. carry a canonical full-policy fingerprint inside the signed annotated tag;
3. run a read-only CI assertion and verify the signed fingerprint before
   publication.

Do not solve a visibility limitation by placing a ruleset-write credential in a
publication job. Customize the canonical policy and repository identity for the
adopting repository, and preserve negative controls for redaction and unexpected
visible actors.

## Changes

| Area        | Change                                                                                        |
| ----------- | --------------------------------------------------------------------------------------------- |
| **Release** | Full ruleset guard runs inside tag creation and emits a signed canonical policy fingerprint   |
| **CI**      | Read-only ruleset validation and signed full-policy attestation replace the impossible repeat |
| **Tests**   | Redaction, unexpected actors, and missing/wrong attestation controls                          |
| **Process** | PDR-0004 and the release checklist document the three-part reusable gate                      |
| **Build**   | Version 0.1.20 → 0.1.21; package metadata, README badge, and changelog links are synchronized |

## Migration Notes

- Existing release tags remain immutable. Do not move v0.1.20 onto this repair.
- Maintainers must create v0.1.21 through `make release-tag`; a manually signed
  tag without the policy fingerprint fails publication.
- CI remains read-only. No new secret or repository permission is required.
- No schema or portable contract changes occur in this release.
