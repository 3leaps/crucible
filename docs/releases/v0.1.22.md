# Release v0.1.22

**A governance release: one Engineering Principle Record on the integrity of
gates.**

## Overview

v0.1.22 adds a single decision record. No schema changes, no capability-token
bump, no contract movement.

EPR-0002 states a durable engineering principle about the class of automated
checks whose passing we treat as evidence — tests, validators, CI assertions,
release checks, schema conformance runs. It generalizes obligations the
organization had already derived independently in three domains, and fixes them
as one cross-cutting principle.

## Highlights

### EPR-0002: Gates Assert on Resolved State and Are Proven Able to Fail

This lane's second Engineering Principle Record. The principle:

> A gate MUST assert on the resolved state the system actually reached, MUST treat
> absent evidence as a failure rather than substituting a value for it, MUST be
> demonstrated capable of failing, and MUST NOT carry an exemption its stated
> coverage does not declare.

Four obligations, each durable and independent of language, framework, and
domain:

- **Assert on resolved state, not requested input.** The assertion is made
  against what the system resolved, produced, or published — not the
  configuration it was handed. Where an input competes to determine an outcome,
  the gate names the outcome and its provenance, so a supplied value that did not
  prevail cannot satisfy a claim that it did.
- **Absence is a verdict, never a placeholder.** When evidence is missing it must
  present as missing and the gate must fail. No layer between the system and the
  gate may substitute a synthetic value for a fact that was never established —
  manufacturing a stand-in is the mechanism by which an unverified system reports
  as verified. Producers are as bound as gates here.
- **A gate is proven by a negative control.** Every gate ships with at least one
  executable test that feeds it a deliberate mutation of the state the gate
  exists to catch. The test must assert that the gate rejects the mutation and
  fail if the gate accepts it. A gate never observed failing is configured, not
  proven; the two are indistinguishable from a green run.
- **An exactness claim admits no undeclared exemption.** A check that claims to
  cover exactly some declared set must reconcile in both directions: every
  encountered element is declared, and every required declared element is
  encountered and judged. Out-of-scope categories are declared in the contract,
  never expressed as a shape-wide skip inside the check.

The record fixes obligations, not tooling. Test framework, mutation approach, and
how provenance is surfaced are local choices expected to change without disturbing
the principle. Adopting repositories link or vendor it per ADR-0002/0003 rather
than forking divergent copies.

Status is **accepted**: ratified on entarch ecosystem-parity assent, with a
conforming reference implementation already landed in a public measurement
harness.

### Relationship to existing records

EPR-0002 generalizes; it does not supersede. The data-pipeline principles
GP-2.2 (no silent skip) and GP-2.4 (later bands re-prove from independent
evidence) remain the data-engineering instances and keep their opt-in domain
scoping. EPR-0001's negative-control requirement remains the supply-chain
instance. PDR-0004 remains the release-publication instance. Where a domain record
states a sharper obligation than this one, the domain record governs within its
domain.

## Migration Notes

- **Adopters of crucible standards:** EPR-0002 is accepted and binds any
  repository whose gates are relied upon to discharge review or authorize a
  release — in practice, all of them. Unlike EPR-0001 there is no qualifying shape
  to check: a repository with no gates has nothing to conform to, and a repository
  with gates is in scope. Record local conformance; do not fork the principle.
- **No schema or contract change:** no capability tokens move in this release.
