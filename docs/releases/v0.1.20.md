# Release v0.1.20

**Governance records: dependency-graph integrity, and the release process that
publishes them.**

## Overview

v0.1.20 is a governance release. It adds two decision records and implements the
one that describes this repository's own release process. No schema changes, no
capability-token bump, no contract movement.

The two records sit at different altitudes on purpose. EPR-0001 states a durable
engineering principle that binds any repository publishing artifacts built from a
resolved dependency graph. PDR-0004 makes a revisable process choice about how
this repository publishes releases — and ships its implementation in the same
change.

## Highlights

### EPR-0001: Published Artifacts Carry an Integral Dependency Graph

This lane's first Engineering Principle Record. The principle:

> Every artifact we publish that is built from a resolved dependency graph MUST
> ship that graph pinned in-repo, enforced at build, continuously audited, and
> held at parity across all distribution surfaces of the same release.

Four obligations, each surface- and language-agnostic:

- **Pin** — every surface producing a distributable artifact commits its
  resolved graph. A pure library published as source for downstream resolution
  is exempt; the obligation attaches to published artifacts, not to libraries
  whose consumers pin.
- **Enforce** — the release build consumes the pin authoritatively, in the
  ecosystem's locked mode. An unenforced pin is worse than none: it presents the
  appearance of control while permitting the drift it appears to prevent.
- **Audit** — every pinned graph is gated for known advisories on every change
  **and on a schedule**, across all pinned surfaces rather than only the primary
  one. Change-triggered scanning alone has a blind spot: advisories arrive
  against unchanged pins, so a quiet repository is otherwise never re-scanned.
  Accepted residual risk is a dated, revisit-conditioned security decision
  record, never a silent ignore.
- **Parity** — when one release ships the same core through several surfaces,
  security-critical shared components must resolve to identical **upstream
  source versions** across every pin. Divergence is a build failure, not a
  warning. The parity set is a **declared, reviewed manifest**: a machine gate
  needs a machine-readable input, or it asserts over the full graph (impossible
  across platforms) or an ad-hoc subset (a blind spot that looks like coverage).

Conformance is demonstrated with a **negative control** — the adopter shows the
locked build _fails_ on a violated pin, not merely that the enforcing flag is
configured.

The record fixes obligations, not tooling. Which locked-mode flag, which advisory
scanner, and how a parity assertion is expressed are local choices expected to
change without disturbing the principle. Adopting repositories link or vendor it
per ADR-0002/0003 rather than forking divergent copies.

Status is **proposed**; it graduates to accepted once a conforming reference
implementation lands and is linked from the record.

### PDR-0004: The Signed Tag Authorizes Publication

The release workflow previously created every GitHub Release as a draft, pending
a manual undraft, under a stated posture of `draft -> sign -> undraft`.

That sequence cannot occur. Signing is what creates the tag object; a tag is
signed before it is pushed, and the push is what triggers the workflow. By the
time CI runs, the tag is already signed. The draft gate guarded a condition that
could not exist, and its omission produced no signal — no failed build, no red
badge — so the repository looked healthy while a release was invisible.

The decision: the signed tag **is** the publication authorization. Pushing a
signed tag is a deliberate, authenticated act that already expresses intent to
release; the undraft click added no authority the signature did not carry. CI now
asserts the signature and publishes.

### Releases Publish From CI, Gated on a Pinned Key

`release.yml` gains a signature-verification job between its quality gates and
release creation. It makes two assertions, both required:

- **Key identity (authoritative).** The tag verifies in an isolated keyring built
  solely from the release keys committed at
  `docs/security/release-signing-keys.asc`.
- **Account linkage (secondary).** GitHub independently reports the tag verified.

The first assertion is what makes the signature an _authorization_ rather than an
attribution. GitHub's verified flag holds for a signature by any key uploaded to
the tagger's account, so on its own it would reduce the publication gate to
"tag-push rights plus a self-uploaded key."

Only after both pass is the release created — directly as published, with
`Latest` set explicitly for stable versions rather than assumed. Prereleases
publish as prereleases and do not take `Latest`.

If the signature is absent, is made by a key outside the pin, or cannot be
verified, the job fails and **no release is created**. This inverts the previous
semantics: an unpublished release now means something went wrong, rather than
something awaits a human.

The authorization is bound to the exact annotated tag object SHA. CI verifies
that object locally and through GitHub, carries its identity into the publishing
job, and reasserts immediately before release creation that the version-tag ref
still resolves to it. Lightweight tags, missing object identities, and ref
changes fail closed.

### Version-Tag Protection Is an Executable Precondition

`make release-tag` now depends on `make release-guard-tag-ruleset`, and the
release workflow repeats the same guard before verifying publication authority.
The guard resolves the live `Tag Publish Protection` ruleset by name and
requires its full publication-policy shape: active protection for only
`refs/tags/v*`; creation, update, deletion, and non-fast-forward protections;
and a sole always-on bypass for organization administrators. Missing access,
malformed responses, extra bypasses, or any policy drift block tagging and
publication.

The check and release workflows also pin every third-party action to a full
commit SHA. The readable versions remain in comments, while upstream movable
tags no longer select the code that executes in the publication path.

### Release-Key Rotation and Expiry

Because publication is gated on committed key material, that material becomes a
release-path dependency. The checklist gains a rotation section: update the pin
by reviewed PR _before_ the first tag signed by a new key. A stale pin fails
closed — red job, unpublished release — which is the correct direction but an
avoidable block.

Key **expiry** is the same failure arriving unprompted. Rotation is chosen;
expiry is not. Track the expiration of the pinned material and refresh ahead of
it, rather than discovering it when a release stops publishing.

### Release Checklist Verifies Published State

Post-release items now confirm the release is non-draft, carries `Latest`, and is
reachable — not merely that a release object was created. The previous items
could tick green while a release was still invisible.

## Migration Notes

- **Adopters of crucible standards:** EPR-0001 is proposed and binds repositories
  that publish artifacts built from a resolved dependency graph. Repositories
  shipping several surfaces over a shared native core are the primary adopters.
  Record local conformance; do not fork the principle.
- **Repositories vendoring this release process:** the draft-then-undraft step is
  gone. If you copy `release.yml`, you must also commit your own release-key pin
  file — publication depends on the tag being signed by a key in it, _and_ on
  that key being registered to the account owning the tagger email. Adopt the
  rotation and expiry checklist items along with the workflow; the pin is a
  release-path dependency, not just a security artifact.
- **Version-tag ref protection is a precondition.** Publication authority now
  flows through `refs/tags/v*`. Adopt the ruleset guard with the documented
  protection and bypass shape rather than relying on an unevaluated repository
  setting (see PDR-0004 §7).
- **Maintainers of this repository:** pushing the signed tag is the last manual
  act. A release found sitting in draft is a defect in the publication path, to
  be fixed in the workflow rather than undrafted by hand.
- **No schema or contract change:** no capability tokens move in this release.
