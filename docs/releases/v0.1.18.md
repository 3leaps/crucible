# Release v0.1.18

**Companion contract for local process telemetry and control.**

## Overview

v0.1.18 adds the portable `process-run/v0` contract — a source-neutral standard
for observing and steering local long-running processes at a deliberately
minimal complexity floor: an append-only NDJSON event stream (the flight
recorder) and a token-gated local control channel with card rendezvous (the
steering wheel). It is the sibling of `data-artifact/v0`: that contract governs
_what was produced_; this one governs _the producing process_. The contract
lands as **proposed** and graduates to accepted on a downstream conforming
implementation.

## Highlights

### Portable Process Run Contract

The `process-run/v0` contract describes how a local long-running process — one
started from a CLI, a service control plane, or a scheduler — exposes what it is
doing (an append-only event stream) and how to reach it (a token-gated local
control channel), so any conforming consumer can observe and steer it without
knowing which producer created it.

The contract is deliberately minimal: files and a local socket, schema-versioned.
It sits below distributed-tracing stacks rather than competing with them —
telemetry ecosystems remain reachable because the event envelope maps to OTLP in
a downstream forwarder, keeping producers OTLP-free.

Design stance: the artifact is the interface for state (a socket exists only
where liveness is required); fail-open emission (telemetry never blocks the work
it describes); local-only transport (no WAN listener, no TLS/PKI in the
producer); lock-and-key control (an invocation-time key plus filesystem
permissions); and payload-open, metadata-governed (producer payloads stay
producer-owned, while the card, event envelope, and control exchange have
closed, versioned semantics).

### Schema Family and Golden Examples

The `schemas/process-run/v0/` family ships the `process-card`, `process-event`,
and `control-exchange` schemas, a host-less `contract.json` entry manifest, a
README, and golden examples (card, control request/response, and an NDJSON event
stream). One family carries one capability token — the three objects are
designed against each other and version together.

### Bridge to Data Artifact

Terminal events may carry `data.artifacts[]` linking a run to its outputs. The
two claims stay independent: a run's own states are the event kinds plus
`running | stopping | done`, while `lifecycle` (`complete | partial`) remains the
artifact's vocabulary. The terminal-event bridge maps between the two namespaces
without conflating them.

### Decision Record

ADR-0006 ratifies `process-run/v0` as a proposed companion portable contract,
recording the vehicle choice (Standard plus ratifying ADR, following the
ADR-0002 / ADR-0004 precedent), the object-named naming rationale, and the six
review-loop items that ride the proposed window.

## Migration Notes

- **Producers:** a conforming producer writes an owner-only process card at
  startup that identifies the run (`run_id`) and points at its event stream
  and/or control socket, and emits schema-versioned NDJSON events. Native
  payloads stay in the producer-owned `data` field.
- **Consumers:** resolve the host-less capability token (`contract: process-run/v0`)
  through a trusted `contract.json` manifest, validate the card before using
  either pointer, and treat a card whose `(pid, started_at)` no longer matches a
  live process as stale.
- **Adopters:** the contract is proposed. The six review-loop items (one contract
  vs three, `pause`/`resume` verbs, in-run rotation, the Windows named-pipe
  profile, heartbeat-cadence guidance, and serve-mode key rotation) are open;
  treat the shapes as stable-intent but subject to hardening before v1.
