# Commit Style

**Canonical URL** (hosted site planned — v0.1.x): `https://crucible.3leaps.dev/repository/commit-style`

Commit message conventions for 3leaps repositories.

## Prerequisites

This standard references repository structures defined in other 3leaps standards:

| Standard                                   | Defines                                       |
| ------------------------------------------ | --------------------------------------------- |
| [agents.md](agents.md)                     | Required files: `AGENTS.md`, `MAINTAINERS.md` |
| [agent-identity.md](agent-identity.md)     | AI contribution attribution                   |
| [makefile-minimum.md](makefile-minimum.md) | Required make targets for quality gates       |

The `@maintainer` references in this document refer to maintainers listed in `MAINTAINERS.md`. See [agent-identity.md](agent-identity.md) for public AI attribution patterns.

## Format

```
<type>(<scope>): <subject>

<body>

<trailers>
```

## Types

| Type       | Purpose                      |
| ---------- | ---------------------------- |
| `feat`     | New feature                  |
| `fix`      | Bug fix                      |
| `docs`     | Documentation only           |
| `refactor` | Code change (no feature/fix) |
| `test`     | Adding/updating tests        |
| `chore`    | Maintenance, deps, tooling   |

## Subject Line

- Imperative mood ("add" not "added")
- No period at end
- 50 characters max
- Lowercase after type prefix

## Body

- Wrap at 72 characters
- Explain what and why (not how)
- Reference issues if applicable

## AI Agent Attribution

When commits are AI-assisted:

```
feat(api): add health endpoint

Implements /health endpoint returning service status.

Changes:
- Add health handler in internal/server/handlers/
- Wire route in server.go
- Add tests

Generated by Claude Sonnet via Claude Code under supervision of @3leapsdave

Co-Authored-By: Claude Sonnet <noreply@3leaps.net>
Role: devlead
Committer-of-Record: @3leapsdave
```

### Required Elements

1. **Generated by** - Model, interface, supervisor
2. **Co-Authored-By** - Agent attribution with `noreply@3leaps.net`
3. **Role** - Public role perspective for the change (see [agent-identity.md](agent-identity.md))
4. **Committer-of-Record** - Human accountable for the commit
5. **Changes** - Bullet list of specific changes

### Committer-of-Record

The Committer-of-Record trailer makes explicit the human accountability for AI-assisted commits.

**Note**: This is a human/AI-readable convention, not recognized by GitHub. GitHub accountability comes from the git author (user.name/email) and the GitHub account that pushed. Committer-of-Record documents this accountability chain explicitly without requiring inspection of git internals.

### Trailer Order

Standardized order for attribution trailers:

```
Co-Authored-By: Claude Sonnet <noreply@3leaps.net>
Role: devlead
Committer-of-Record: @3leapsdave
```

The `Generated by` line appears before trailers, separated by a blank line.

### 3leaps vs FulmenHQ Format

**3leaps tools** (this standard):

```
Generated by Claude Sonnet via Claude Code under supervision of @3leapsdave

Co-Authored-By: Claude Sonnet <noreply@3leaps.net>
Role: devlead
Committer-of-Record: @3leapsdave
```

**FulmenHQ tools** (extended details):

```
Generated by Pipeline Architect ([Codex CLI](https://github.com/openai/codex)) under supervision of @maintainer

Co-Authored-By: Pipeline Architect <noreply@3leaps.net>
Role: entarch
Committer-of-Record: Jane Doe <jane@example.com> [@maintainer]
```

| Element           | 3leaps Baseline   | FulmenHQ Full            |
| ----------------- | ----------------- | ------------------------ |
| Email in CoR      | No (just @handle) | Yes (full email)         |
| Human name in CoR | No                | Yes                      |
| Interface URL     | No                | Yes (linked)             |
| Role trailer      | Yes (required\*)  | Yes (required)           |
| Agent identity    | Model name        | Repository-specific role |

\* Required for code-substantive, security-sensitive, and data-integrity repositories.

## Examples

### Simple fix

```
fix: correct exit code for missing config

Was returning 1 (general error), now returns 3 (config error)
per exit code standard.
```

### Feature with AI attribution

```
feat(cli): add --verbose flag

Adds verbose output mode for debugging.

Changes:
- Add --verbose/-v flag to root command
- Increase log level when enabled
- Update help text

Generated by Claude Sonnet via Claude Code under supervision of @3leapsdave

Co-Authored-By: Claude Sonnet <noreply@3leaps.net>
Role: devlead
Committer-of-Record: @3leapsdave
```

## Security-Sensitive Repositories

Repositories handling authentication, cryptography, or untrusted input have additional commit message requirements to prevent inadvertent disclosure of vulnerability details.

See [secure-commits.md](secure-commits.md) for:

- Repository designation process
- Functional vs contextual language rules
- Restricted keywords and alternatives
- Enforcement mechanisms (hooks, CI, review)
