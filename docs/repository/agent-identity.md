# AI Attribution

**Canonical URL** (hosted site planned — v0.1.x): `https://crucible.3leaps.dev/repository/agent-identity`

Public attribution guidance for AI-assisted contributions in 3leaps
repositories.

## Scope

This standard defines what public commit and review metadata should say when AI
tools assist with a change. It does not define private account provisioning,
runtime identity, chat routing, session memory, or internal coordination
workflows.

## Principles

1. **Human accountability is explicit.** A human maintainer remains responsible
   for accepting the change.
2. **The AI role is described generically.** Use a public role slug such as
   `devlead`, `devrev`, `secrev`, `qa`, `releng`, or `docs`.
3. **Metadata is durable.** Commit messages, PR descriptions, and review
   comments should be written as public records.
4. **Private context stays out.** Do not include private task identifiers,
   private channel names, internal filesystem paths, customer details, secrets,
   or non-public repository names.

## Commit Attribution

Use the repository's commit style plus an AI attribution block when an AI tool
materially assisted the change:

```text
docs(example): clarify validation guidance

Updates the validation guidance with a focused example and expected failure
mode.

Generated by <Model> via <Interface> under supervision of @<maintainer>

Co-Authored-By: <Model> <noreply@3leaps.net>
Role: <role>
Committer-of-Record: @<maintainer>
```

Use a bare role slug in `Role`, not a bot username or internal roster label.

## Role Labels

Recommended public role labels:

| Role      | Use When                                          |
| --------- | ------------------------------------------------- |
| `devlead` | Implementation or repository maintenance          |
| `devrev`  | Code review or change review                      |
| `secrev`  | Security review or security-sensitive guidance    |
| `qa`      | Testing, verification, or quality-gate work       |
| `releng`  | Release process, changelog, or packaging work     |
| `docs`    | Documentation-only changes without a narrower fit |

Repositories may document additional public role labels when the label is useful
to external readers.

## PR And Review Text

PR descriptions and reviews should describe:

- What changed.
- How it was verified.
- Which public role perspective was applied, when useful.

They should not include:

- Private planning context.
- Internal escalation routes.
- Private tool, service, or repository names.
- Local-only paths or machine-specific setup details.

## Examples

Good:

```text
Generated by GPT-5 via Codex under supervision of @maintainer

Co-Authored-By: GPT-5 <noreply@3leaps.net>
Role: devlead
Committer-of-Record: @maintainer
```

Avoid:

```text
Role: private roster label
Task: internal task identifier
Coordination: internal thread name
```
